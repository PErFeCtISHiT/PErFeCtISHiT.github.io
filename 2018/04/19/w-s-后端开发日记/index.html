问题及解决办法

映射静态资源文件

​ 发生的原因是项目打包、前后端项目分离后项目需要映射静态资源文件到某个resource里面，不使用项目原来的resource。在项目启动时，会自动去对应的路径下查找静态资源，例如，在访问web url时，172.0.0.1:8080/test，便会去以下文件夹查找： /static 、/public、/resources、/META-INF/resources， spring boot给我们了一个配置的简便办法，否则可以使用config类或者xml去进行配置。

​ 在resources文件夹下，新建一个名为application.properties的文件，并输入：

spring.resources.static-locations=file:C:/java/SE/ImgAnnotator/annotator/

这里我的文件路径是windows系统下的某个文件夹，如果是项目的某个地方，则将file:改成classpath:，这样，在项目启动时，会自动去对应的路径下查找静态资源，例如，在访问web url时，172.0.0.1:8080/test，便会去对应路径下查找需要的test响应。

项目后端架构

​ 迭代二硬性要求不许使用数据库，使用了json进行数据的持久化，但是为了将来程序使用数据库后减少不必要的修改，将与数据的操作都内聚在dao层，减少了service与dao之间的耦合，将来只需要修改dao包里的方法，便可以在使用数据库的情况下继续正常使用service里面的服务。eneity则是正常的数据持久化对象。util里面包含了此次项目可能会用到的工具类方法。工具类都是抽象的，并且从sonar那里学到了一招：

private DateHelper() { throw new IllegalStateException("Utility class");}

在抽象类上加上抛出异常的构造函数，这样如果你粗心创建了一个抽象类，便会抛出异常，提醒你这个类不能被实例化。直接调用它的静态方法。config包基于shiro架构，配置了shiro的身份认证、权限管理。是spring boot的特色之一，使用配置来代替xml config，方便好用。controller包是与前端交互的核心，包含了所有接口，响应前端请求，调用service进行处理，最后将结果返回给view，是view与service的媒介。

代码测试

​ 单元测试是你一个人写有质量代码的保障，在做大型，复杂项目时，单元测试往往可以发现更多的问题，比手动测试更加高效，而且随改随测，一次性写好一个测试代码，可以受用到项目交付。集成测试则是与前端交付的保障。

sonar代码评价系统

​ sonar是一款看起来“很舒服”的软件，（相比于Jenkins。。。。）sonar用来评估代码质量，提高你写代码的水平，只要有一定的基础，然后每次写完代码后都用sonar去评一下，修正里面的所有警告，进步的速度是十分显著的。sonar与maven配套使用，只需在终端项目路径下输入：

mvn sonar:sonar，便可以评估你的代码。sonar还有统计功能，统计你写代码出错的“能力”，累计你的“债务”。

Jenkins

​ 持续集成时必要软件吧，但是功能太复杂，还没有研究透彻。。将来再补在这里吧。。。（界面好丑）

json

​ json是一套存储数据的约定，用键值对的迭代嵌套了存储复杂的数据，理论上来说json也属于一种数据库，又是前后端交互的媒介。java中比较好用的json工具包是org.json，令我很不舒服的是google.json，org.json对object和array的操作可以当作一般map与array一般，大同小异，然而google.json却存在很多问题，因此建议初学使用json的时候选好工具包，不然使用了一段时间后再更换，在整个项目中修改会十分痛苦。
