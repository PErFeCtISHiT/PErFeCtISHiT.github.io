os进程调度

Posted on 2018-05-19 | In 基础

实验要求

设计调度算法对进程进行调度，考虑cpu个数，资源占用。提供一块命名为freeMemory的内存。

实现

参考了提供的贪心算法，贪心算法的实现原理是每个时间片进程调度时清空所有资源，从任务列表中选择一个任务进行调度，此算法对于内存读写很不友好，每次进程调度都需要读写资源，读写任务，cpu内运行的任务切换次数也比较多，优点是任务的等待时间较短，我在设计算法时，便基于内存读写与cpu切换，希望尽量减少内存读写次数，cpu切换次数。

code说明

算法是fcfs+最大资源数优先，schedule类的变量声明了一些内存中的偏移，如pcb，资源、cpu，可以通过最新的任务id来得到任务的总个数，每个任务有自己的数据结构，记录了任务的到达时间，剩余时间等信息。在ProcessSchedule时，首先根据内存还原cpuOperate的状态，找到上一时刻cpu内运行的进程，如果进程结束，则选择进程队列中需要资源数最大并且资源均可以使用的进程运行，为其分配资源，否则继续运行之前的进程，这样cpu的切换只发生在进程运行结束时（非抢占），内存的读写也仅存在于读取cpu中的进程，切换进程。实际检测发现内存的写次数为2572，是贪心算法的1/50，cpu切换次数是20，是贪心算法的1/2，由于需要还原现场，使用了较多的读取内存操作，内存的读取次数增加了一倍，最后，满意度增加了一倍。

总结

fcfs+最大资源数优先，每个进程非抢占式的独占cpu，优先选择使用资源多的进程去cpu执行，可以尽量减少资源零头，参照了可变分区存储管理的思想，算法最终满意度比较长，但是实现了减少内存读写，减少cpu切换。

Posted on 2018-05-15 | In 开发

jpa+hibernate+spring boot

使用中的问题及解决

报错信息的利用

在hibernate的entity中有几处表的关联写错了，然后spring boot运行报错，然而exception给的信息里没有提供是这个entity出错，要仔细在报错里面找，才可以找到。

hbm2ddl

在每次运行项目时，hbm2ddl会检查entity，去数据库中修改表，执行时间很慢，因此在property里面将其设置为none就好啦

linux服务器数据库配置

参考网上教程，然而一直配不好，发现是ubuntu里的防火墙配置的问题，而且有不同的版本，，，在配置完数据库服务后，查看端口，如果有端口并且监听的ip地址是全局的，还连不上数据库，一般就是防火墙的问题了。

建表

根据去年经验，先建表后用idea的hibernate插件自动生成hbm，然而spring boot并不需要hbm.xml，只需要在entity里面好好设计表，然后会通过hbm2ddl修改数据库，，因此正确的流程应该是写entity，自动生成表，这样表也更加严格正确。

jpa

这次是第一次用到jpa，持久化层的一个api，看起来功能强大，写好对数据库持久化的接口以后就可以自动装配。。。

关联映射

通过在entity里面建立表属性，进行表与表之间的one to many，many to one，many to many关联映射，设置级联关系（一般为merge，更新和新增时会对关联的表进行对应操作），ssh框架中controller返回对象时使用了json工具来将对象转换为json，因此需要@jsonignore来对表内的对象进行约束，不返回这些对象属性（否则会无限递归，跑崩线程）

os_io

Posted on 2018-05-15 | In 基础

用OS实现IO

说明

代码参考了orange’s的源码，从chapter7/n开始进行改动，在keyboard.c里增加了mode全局变量，用来表示与查找模式有关的状态并在console.c里使用了这个变量，在console.h里增加了几种颜色变量，修改了keyboard.c,console.c,tty.c,tty.c里新增了根据时钟刷新console，search接口，tab键的判断。console.c里新增了search逻辑，清屏逻辑，tab逻辑，修改了退格键的逻辑来实现精细处理，keyboard.c新增了sec判断，回车判断来实现search的状态转换

流程

这里跳过了boot->loader,loader->kernel，orange’s上有源码，原理参考第三章，进入内核模式，开启键盘中断，开启时钟中断，运行tty进程，taskA,B,C（并没有用到），tty进程负责应并处理键盘输入，随时读取键盘输入，并且输出到console上，这里比较关键的几个文件有console.c,tty.c,keyboard.c,console.h,tty.h,keyboard.h,global.h，.c文件负责实现具体的功能，.h文件声明.c中用到的数据结构，tty,console,keyboard都有一个结构体。global.h文件声明了一些全局变量。

细节

keyboard.c

键盘的代码中分析了键盘中断的端口代码，将结果告诉tty，tty根据不同的按键进行不同的操作，这里键盘响应都是以按下某个键作为开始响应，而抬起大多数没有考虑，也符合常理。（考虑make，忽略break）。

tty.c

tty接收键盘传来的指令，用console.c显示，orange’s中支持多个tty，这里只开了一个。

console.c

console作为显示，需要显示东西时，可以先声明一个char*，指向当前console所显示的位置，然后修改其后面的内容，显示到console上，再修改cursor，指向当前输入。

文件组织结构

orange’s的源码参考了c语言的组织方式，有lib，有include，参考这个结构，在对应的lib里加自己需要的lib函数，在include里面加需要的头文件，由于是在自己的os上运行，所以没有<stdio.h>等等的c语言头文件，所以在写c语言代码时，只能使用自己写的函数，而且目前还没有尝试过debug，还好问题不是很大……

makefile

参考orange’s，完善了makefile，clean，buildimg，make，run。makefile是真的好用。。。。64位linux的makefile需要注意在所有的gcc指令后面添加-m32生成32位.o文件，在ld指令后面添加-m elf_i386。

fat12 实验笔记

实验要求读取一个1.44Mfat12格式的软盘里的文件，实现ls，cat，count指令，具体实验内容并没啥亮点，在cpp、混编中遇到许多问题。。。。（开发环境：ubuntu64，clion，gcc）如果你也在用linux64位，，那么这里面的2/3的问题也可能发生。。。。（每时每刻都想把自己的系统卸了装32位的）

code:https://paste.ubuntu.com/p/mRd3KKB5VK/ （没加汇编，自行脑补）

  1. new or instance

    new或是声明一段空间，在平时使用的时候并没有太过在意，但是由于编译器版本的问题，在gcc，g++这种版本的编译器下使用new有时候多级递归会产生越界（有时候又没有。。。）会经常遇到Error: free(): invalid next size (fast)因此这里建议尽量用声明内存空间并自行管理来代替new。

  2. code中使用了树的结构代表file文件系统，这里自然而然的想法是使用new，在后面尝试着将new转换成声明，但是没有改好，也不影响程序运行，因此没有继续解决。。。
    S

  3. 非标准扩展函数、c++11以上函数、标准c/c++函数

    以atoi为例，这也是这次代码中遇到的问题，在我需要使用atoi这个函数时，发现linux环境里并不包含这个函数，后来使用sprintf来代替它，在clion中编译链接执行没有任何问题，但是使用gcc联编时，找不到sprintf这个函数，以及nullptr等c++11的新特性，甚至找不到fread函数。。。。前面两个还可以想办法换方法，但是fread我不知道用什么去替代，后面查阅资料发现gcc的命令-std=gnu++0x可以解决这类问题，因此程序中建议不要使用非标准扩展函数，可以使用c++新特性，标准c/c++函数。

  4. extern函数，实验中有cpp与nasm联编的情况，在函数声明时需要加extern “C”来代表它的定义是一段c的代码，如果是c与nasm 联编，则只需extern，甚至用gcc自行链接。

  5. 分配空间不足引起的错误：‘Stack smashing detected’,遇到这个问题真的是特别奇怪，程序逻辑没有任何问题，但是gcc有一种缓存溢出的保护机制，用来防止溢出性的攻击，然后我把我的数组开大了一点点，还是报错，开大了10倍，错误解决。。。。。。

  6. bits/c++config.h: 没有那个文件或目录，如遇到缺少 bits/c++config.h的情况，输入以下命令安装gcc编译相关库 sudo apt-get install gcc-4.7-multilib g++-4.7-multilib
  7. i386:x86-64与i386 在64位操作系统下要注意的问题，记得生成32位的两个可执行文件，，，最好是32位操作系统啦，64位也可以的，差点卸载重装。。。
  8. 64位linux制作.img软盘，制作方法与32位操作系统一样，但是，当做好一个1.44M fat12后，试图往里面加文件，
    在挂载到一个目录下添加文件后，发现里面的目录区跟fat12的根本不一样，fat12的目录区32个字节代表一个目录，这里却是64位代表一个，而且当我卸载了软盘，重新挂载后，找不到原来那几个文件了，这里感觉64位操作系统制作软盘很坑很坑，最后用windows winimg制作了一个软盘拷到linux系统下才得以解决。。

imgannotator标注系统后端开发的过程—。

项目结构图

项目结构图日常修改，目前的结构图如下（服务器客户端两个项目并行，在idea的2个moudle下，注意新建一个moudle并且是maven项目时，为它新建一个pom.xml并且去继承外层的pom.xml。不然会发生找不到library的问题。

问题及解决办法

映射静态资源文件

​ 发生的原因是项目打包、前后端项目分离后项目需要映射静态资源文件到某个resource里面，不使用项目原来的resource。在项目启动时，会自动去对应的路径下查找静态资源，例如，在访问web url时，172.0.0.1:8080/test，便会去以下文件夹查找： /static 、/public、/resources、/META-INF/resources， spring boot给我们了一个配置的简便办法，否则可以使用config类或者xml去进行配置。

​ 在resources文件夹下，新建一个名为application.properties的文件，并输入：

spring.resources.static-locations=file:C:/java/SE/ImgAnnotator/annotator/

这里我的文件路径是windows系统下的某个文件夹，如果是项目的某个地方，则将file:改成classpath:，这样，在项目启动时，会自动去对应的路径下查找静态资源，例如，在访问web url时，172.0.0.1:8080/test，便会去对应路径下查找需要的test响应。

项目后端架构

​ 迭代二硬性要求不许使用数据库，使用了json进行数据的持久化，但是为了将来程序使用数据库后减少不必要的修改，将与数据的操作都内聚在dao层，减少了service与dao之间的耦合，将来只需要修改dao包里的方法，便可以在使用数据库的情况下继续正常使用service里面的服务。eneity则是正常的数据持久化对象。util里面包含了此次项目可能会用到的工具类方法。工具类都是抽象的，并且从sonar那里学到了一招：

private DateHelper() { throw new IllegalStateException("Utility class");}

在抽象类上加上抛出异常的构造函数，这样如果你粗心创建了一个抽象类，便会抛出异常，提醒你这个类不能被实例化。直接调用它的静态方法。config包基于shiro架构，配置了shiro的身份认证、权限管理。是spring boot的特色之一，使用配置来代替xml config，方便好用。controller包是与前端交互的核心，包含了所有接口，响应前端请求，调用service进行处理，最后将结果返回给view，是view与service的媒介。

代码测试

​ 单元测试是你一个人写有质量代码的保障，在做大型，复杂项目时，单元测试往往可以发现更多的问题，比手动测试更加高效，而且随改随测，一次性写好一个测试代码，可以受用到项目交付。集成测试则是与前端交付的保障。

sonar代码评价系统

​ sonar是一款看起来“很舒服”的软件，（相比于Jenkins。。。。）sonar用来评估代码质量，提高你写代码的水平，只要有一定的基础，然后每次写完代码后都用sonar去评一下，修正里面的所有警告，进步的速度是十分显著的。sonar与maven配套使用，只需在终端项目路径下输入：

mvn sonar:sonar，便可以评估你的代码。sonar还有统计功能，统计你写代码出错的“能力”，累计你的“债务”。

Jenkins

​ 持续集成时必要软件吧，但是功能太复杂，还没有研究透彻。。将来再补在这里吧。。。（界面好丑）

json

​ json是一套存储数据的约定，用键值对的迭代嵌套了存储复杂的数据，理论上来说json也属于一种数据库，又是前后端交互的媒介。java中比较好用的json工具包是org.json，令我很不舒服的是google.json，org.json对object和array的操作可以当作一般map与array一般，大同小异，然而google.json却存在很多问题，因此建议初学使用json的时候选好工具包，不然使用了一段时间后再更换，在整个项目中修改会十分痛苦。

思想

使用两个数组表示fib数列的两个数的每一位。

global main

main:

注意事项

  1. mov指令，寄存器向地址mov，mov长度为寄存器长度，地址向寄存器mov，长度可以手动指定，也可以不指定。movzx，高位补0
  2. .len equ $ - add 可以用来代表一个串的长度
  3. div指令，eax，edx必须置0，否则会报浮点数错误
  4. 地址空间申请是连续申请，因此申请100个byte的空间最好多申请10个，防止后续写入破坏原地址内的值
  5. 本次实验没有用到push指令，64位操作系统下的push指令要使用rax，与标准32位nasm略有区别
